# MVP5 Settings 收口记录（2026-04-08）

## 1. 文档目的

这份文档只服务当前 MVP5 收口，不追求把 Settings 做成完整配置中心。
目标只有两个：

1. 让答辩主链需要的配置真正生效。
2. 把当前明显是假功能、语义不准或后端根本没吃的字段收掉，避免继续误导。

本轮优先级顺序：

1. 真实生效
2. 明确“重启后生效”
3. 去掉假功能
4. 后置复杂能力

---

## 2. 当前收口原则

### 2.1 允许保留的字段

只有三类字段允许继续留在 Settings 页面里：

1. 后端当前已经真实在吃，或者这一轮明确要接上的字段。
2. 虽然不能热更新，但可以明确标注“保存后重启生效”的字段。
3. 虽然暂时不做，但有非常明确的后续实现计划，且页面上会显式标注“本轮未生效”。

### 2.2 本轮必须清掉的字段

以下几类字段不能继续以“正常可配”的姿态留在页面里：

1. 前端能保存，但后端完全不读取。
2. 字段名看起来很专业，但当前没有真实口径支撑。
3. 一旦用户追问“它具体怎么生效”，当前答不上来或实现路径明显不成立。

---

## 3. Settings 字段分层结论

## 3.1 第一层：本轮优先做成真生效

### A. 基础设置

#### 1) 系统语言 `app_language`
保留。
理由：实现简单、观感直接、能明显提升演示完成度。
生效方式：热更新即可，前端切换语言包并同步保存到后端。

#### 2) HTTP 端口 `http_port`
保留。
理由：这是明确的系统入口配置，后续如果前端接“启动后端”流程，也必须有这个字段。
生效方式：冷启动配置，保存后重启生效。
页面语义必须明确写成“下次启动生效”或“重启后生效”。

### B. AI 设置

#### 3) AI Provider / Model / API Key / 输出语言
保留。
理由：这是主链真实配置，不应该继续是假 UI。
当前最小集合：`mock`、`gemini`。
后续可扩：国内 provider 单独拆任务，不和本轮收口绑死。
补充语义：
- `ai_language` 本轮按冷启动配置收口，保存后重启生效。
- 原因不是技术上绝对做不到热更新，而是它直接参与底层固定 prompt 的渲染；
  运行中切换会让前后两批分析任务的输出口径变化，演示和排障时都不够干净。

#### 4) Prompt 列表 + Active Prompt
保留，而且必须真生效。
理由：这块已经和主链分析语义直接相关，不能只是“本地编辑器摆设”。
当前要求：
- Prompt 列表可加载
- 可编辑
- 可新增/删除
- 可切 active prompt
- 主链分析读取 active prompt
补充语义：
- `prompt 列表`
- `active prompt`
- `ai_language`
  这三项本轮统一按冷启动配置收口，保存后重启生效。
- 这样做的原因很直接：它们都会参与最终分析 prompt 的渲染。
  既然主链分析是异步批量任务，运行中切换这几项，会让不同批次任务吃到不同 prompt 口径，
  对答辩展示和问题排查都不友好。
- 所以本轮不承诺运行中热切 prompt，不做“保存后立即影响后续 batch”的语义。

### C. 告警集成

#### 5) Webhook 渠道（本轮只保留飞书）
保留，而且要真生效。
本轮保留字段：
- enabled
- name
- webhook_url
- secret
- threshold

本轮移除字段：
- 通用厂商下拉（不再假装支持多平台）
- payload template JSON（先去掉）

说明：
当前主链已经验证过飞书真实发送、可选签名、critical trace 端到端触发，所以这块最值得直接接回 Settings。

### D. 内核/主链控制

#### 6) 工作线程数 `kernel_worker_threads`
保留。
理由：这是明确的后端核心参数。
生效方式：冷启动配置，保存后重启生效。

#### 7) Trace 聚合/打包相关阈值
建议新增或收拢成真正的“主链配置组”。
建议包含：
- 是否依赖 `trace_end` 作为完成信号
- token 打包限额
- span 容量限额
- flush / dispatch 时间阈值
- 水位阈值配置

说明：
这批字段比“自适应模式”更贴近当前真实后端链路，也更容易解释它们到底在控制什么。

---

## 3.2 第二层：可以保留，但只允许按“重启生效”或“后置实现”展示

### 8) 日志保留天数 `log_retention_days`
值得保留，但不能只停留在输入框。

这类字段真正落地至少要解决三件事：
- 谁来定时扫描
- 按什么规则删（按天、按空间、还是两者一起）
- Trace 主记录、Span、AI 分析是否一起删

所以这不是“顺手补一个 if”就结束的字段，而是一套清理策略。
本轮建议：
- 文档先收口为“值得做”
- 如果时间不够，不要继续把它当已经生效的功能展示

### 9) 最大磁盘占用 `max_disk_usage_gb`
同上。
它只能和 retention 一起讨论，不能孤立存在。
因为“按空间删”本身就需要和“删哪些数据、删到什么程度、何时触发”绑在一起。

---

## 3.3 第三层：本轮建议直接移除

### 10) 自适应模式 `kernel_adaptive_mode`
移除。
理由：当前项目没有真正完成“动态调 batch / 动态调线程 / 动态调阈值”的整套闭环。
继续留着只会把用户带到错误预期。

### 11) 自适应最大批大小 / 自适应相关文案
移除。
理由同上。

### 12) 刷新间隔 `kernel_refresh_interval`
建议移除当前 Settings 页面展示。
理由：这个名字非常泛，用户很难知道它是在控制 flush、tick、采样还是别的节奏。
如果后端确实有真实的某个定时器阈值要暴露，应该换成语义更明确的名字重新入场。

### 13) 通用 webhook 厂商选择
移除。
本轮只做飞书，不继续保留 generic/dingtalk/slack 这种没有完成联调的展示口径。

### 14) Payload Template JSON
移除。
理由：当前演示链路里它会显著增加复杂度，但不会增加核心价值。

---

## 3.4 第四层：后置复杂能力

### 15) AI 自动降级 `ai_auto_degrade`
后置。
如果要做，第一版建议语义非常简单：
- 主 provider 调用失败
- 走 fallback provider/model
- 只处理真正的调用失败，不先引入复杂策略

### 16) AI 熔断 `ai_circuit_breaker`
后置。
如果要做，第一版建议语义也要收窄：
- 连续失败超过阈值
- 暂时关闭 AI 分析
- 主链继续接收、聚合、存储 trace
- cooldown 后再尝试恢复

### 17) 国内 provider / 新 SDK 接入
后置单独拆任务。
不要和本轮 Settings 收口混在一起推进。

## 4. IO 缓冲区字段的结论

## 4.1 当前现状

当前仓库里 `kernel_io_buffer` 只存在于配置结构、SQLite 配置表和前端 Settings 表单里。
它还没有进入真正的后端运行时链路。

也就是说，当前这个字段是“有存档、有 UI、但没有真实控制面”的状态。

## 4.2 这字段当前最大的问题

`IO 缓冲区` 这个名字很容易误导。
因为它可能让人以为它表示：
- socket 接收缓冲区
- HTTP body 暂存区
- Trace 聚合内存上限
- 线程池排队内存上限
- 整个进程可用内存预算

但这几个东西根本不是一回事。

如果在语义没钉死之前就把它接进背压逻辑，后面一定会把“网络 IO 缓冲”和“业务对象堆内存”混在一起。

## 4.3 本轮建议

本轮建议把 `kernel_io_buffer` 从 Settings 页面移除。
理由不是它不重要，而是它现在没有真实且单一的语义。

如果以后真要做，建议先改名，再实现，不要沿用现在这个名字硬接。
更合适的方向有两个：

### 方向 A：做成“聚合态内存保护预算”
例如改成：
- `trace_memory_budget_mb`
- `ingest_memory_guard_mb`

它表达的是：
“给 Trace 聚合态、待分发对象、相关缓存留一个进程内保护预算。”

### 方向 B：做成“网络接入缓冲预算”
这个方向更偏底层网络栈，但当前项目主链并没有把这层预算做成独立、可观测、可控的对象，所以本轮不建议选它。

## 4.4 如果以后真的把它接进背压，正确姿势是什么

不是“快到限额了，就把阈值再往上抬”。
那是在拖延保护时机，相当于系统越危险，你越放宽准入，方向反了。

正确做法应该是：

1. `io_buffer / memory_budget` 作为一个额外的保护信号，而不是去改写原来的水位定义。
2. 原有水位仍然看主链真实积压，例如：
   - active sessions
   - total buffered spans
   - pending tasks
3. 预算占用率只作为“附加升级条件”：
   - 例如 80% 进入 warning / overload 候选
   - 90% 强制 overload
   - 95% 或更高直接 critical，拒绝新 trace
4. 恢复时要带回滞，不要一降到 89% 就立刻解除，避免状态抖动。

换句话说：
它应该“抬高压力等级”，而不是“抬高可接受上限”。

## 4.5 为什么不能直接拿它当水位阈值本体

因为现有背压判断已经基于真实业务积压对象：
- Trace 会话数
- Span 数
- 线程池待处理任务数

这些指标能直接回答“系统现在有没有被压住”。

而 `io_buffer` 更像一个额外的资源保护闸门。
它适合做最后一道兜底，而不是替代原有水位体系。

## 5. 当前建议的页面最终结构

### 5.1 保留分组

1. 基础
- 系统语言
- HTTP 端口（重启生效）

2. AI
- provider
- model
- api key
- 输出语言（重启生效）
- prompt 列表（重启生效）
- active prompt（重启生效）

3. 告警
- 飞书 webhook
- enabled
- name
- url
- secret
- threshold
- test send

4. 内核 / 主链
- worker threads（重启生效）
- trace_end 完成语义
- token limit
- span limit
- flush/dispatch 时间阈值
- 水位阈值

### 5.2 本轮移除

- 自适应模式
- 自适应相关字段
- 通用 webhook vendor
- payload template JSON
- `kernel_io_buffer`
- 当前语义不明确的 refresh interval

---

## 6. 当前收口顺序建议

1. 先改 Settings 文案和分组，把假字段收掉。
2. 再把飞书 webhook 从临时启动参数切回 Settings 真生效。
3. 再收 AI provider / prompt / active prompt。
4. 最后再决定是否补一批内核主链参数。

这样做的原因是：
先让页面语义干净，再接后端生效链路，问题更容易定位。
