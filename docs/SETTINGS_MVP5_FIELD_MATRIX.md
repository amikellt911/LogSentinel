# MVP5 Settings 字段评估矩阵（2026-04-08）

## 1. 这份文档的目的

这份文档是对 `docs/SETTINGS_MVP5_CLOSURE.md` 的补充和纠偏。

这里不再用“有没有实现”作为唯一裁剪标准。
而是改成用下面四个维度来判断一个 Settings 字段要不要保留、要不要本轮实现：

1. 业务/演示收益高不高
2. 当前实现成本高不高
3. 和现有后端架构是否契合
4. 字段语义是否已经稳定、能不能讲清楚

所以这份文档的核心不是“删未实现功能”，而是：

哪些字段值得现在做
哪些字段值得先做简化版
哪些字段值得保留为规划项
哪些字段确实应该先下线

---

## 2. 评估维度定义

### 2.1 价值（Value）

看这个字段对当前 MVP5 的作用大不大。

高：
直接影响答辩主链、联调闭环、用户观感，或者是用户一定会问到的关键配置。

中：
有一定价值，但不是这轮主链必须项。

低：
即使没有它，也不影响当前交付质量。

### 2.2 实现成本（Cost）

不是只看代码量，而是看它会不会牵扯：
- 新状态机
- 新线程模型
- 新数据表/新定时器
- 新协议适配
- 新联调链路

### 2.3 架构契合度（Fit）

看这个字段是不是刚好能挂到现有主链上。

契合度高：
现有仓库里已经有配置结构、后端消费点、前端表单或真实运行态支撑。

契合度低：
当前只是 UI 有个输入框，但后端链路里根本没有对应的控制面。

### 2.4 语义稳定性（Semantic Clarity）

这是最容易忽略的一条。

不是字段名字看起来专业就算语义稳定。
真正稳定的标准是：
- 这个字段到底控制什么对象
- 在什么时刻生效
- 对系统行为的影响是什么
- 如果别人问“为什么这样配”，能不能说清楚

只要这四个问题答不上来，就说明它语义还没钉死。

## 3. 字段矩阵总览

## 3.1 建议“本轮直接做”的字段

### 1) 系统语言 `app_language`
- 价值：高
- 成本：低
- 契合度：高
- 语义稳定性：高
- 本轮策略：直接做

原因：
这是最典型的“小成本、高观感、高确定性”字段。实现不复杂，但答辩时很容易体现“系统完成度”。

当前先钉住的最小语义：

1. `app_language` 是前端界面语言配置，属于热更新字段。
2. 用户保存后，页面文案可以立即切换，不需要重启服务。
3. 后端对这个字段的职责主要是持久化，保证页面下次打开时还能恢复用户上次选择。
4. `app_language` 只控制 UI 文案，不控制 AI 分析结果的输出语言。
5. AI 输出语言由 `ai_language` 单独控制，不能和界面语言混用。

### 2) HTTP 端口 `http_port`
- 价值：中高
- 成本：低
- 契合度：高
- 语义稳定性：高
- 本轮策略：做，但明确为“重启后生效”

原因：
这个字段天然属于系统启动参数，不需要强行热更新。只要前后端语义写清楚，就值得保留。

当前先钉住的最小语义：

1. `http_port` 是冷启动配置，不做运行时热切换。
2. 用户点击保存后，不会立刻修改当前正在监听的端口。
3. 这个配置在“下次启动”或未来的“保存并重启”流程里生效。
4. Settings 页面必须明确标注“重启后生效”，避免用户误以为保存后当前连接会自动切端口。

### 3) AI Provider / Model / API Key / 分析语言
- 价值：高
- 成本：中
- 契合度：高
- 语义稳定性：高
- 本轮策略：直接做

原因：
这批字段和当前主链 AI 分析直接相关，而且已有配置表、前端表单和主链消费路径，属于标准真配置。

当前先钉住的最小语义：

1. `provider` 控制当前主链默认走哪个 AI 提供商。
   第一版最小集合先收口为：
   - `mock`
   - `gemini`
2. `model` 是当前 provider 下默认使用的模型名。
   它不是全局统一语义，而是交给当前 provider 自己解释。
3. `api_key` 是 provider 级密钥配置。
   - `mock` 可以不依赖真实 key
   - 真实 provider 没有 key 时，不应正常发起真实调用
4. `ai_language` 控制的是 AI 分析结果的输出语言，不是前端界面语言。
5. `ai_language` 的实现方式，不是单独改底层算法，而是通过“系统底层固定 prompt”注入语言约束，
   明确要求模型输出中文或英文。
6. 这组配置只影响之后的新分析任务，不回溯重算已经落库的历史 trace 分析结果。

### 4) Prompt 列表 + Active Prompt
- 价值：高
- 成本：中
- 契合度：高
- 语义稳定性：高
- 本轮策略：直接做

原因：
Prompt 不是装饰项，而是 AI 主链真正的输入参数来源之一。这块如果继续半真半假，会直接影响系统可信度。

当前先钉住的最小语义：

1. Prompt 采用“两层 prompt”结构：
   - 系统底层固定 prompt
   - 用户可维护的业务 prompt
2. 系统底层固定 prompt 不开放给用户编辑。
   它主要负责：
   - 输出格式约束
   - 风险等级口径
   - 系统角色与安全护栏
3. Settings 页面维护的是“业务 prompt 列表”，不是整个系统 prompt。
4. 运行时拼接方式是：
   - 系统固定 prompt
   - 当前激活的业务 prompt
   - trace / runtime 上下文
5. 第一版同一时刻只允许一个“全局 active 业务 prompt”参与主链分析，
   先不扩展成按服务、按场景分别选不同 prompt。
6. 这样做的目的，是既保留业务可调性，又避免用户误改底层系统 prompt，
   把输出结构或系统约束搞坏。

### 5) 飞书 Webhook（enabled / name / webhook_url / secret / threshold）
- 价值：高
- 成本：中
- 契合度：高
- 语义稳定性：高
- 本轮策略：直接做

原因：
真实联调已经跑通，当前唯一缺的是把“临时启动参数直连”收回到 Settings 真生效。

当前先钉住的最小语义：

1. 本轮 Settings 里的告警渠道只支持飞书，不继续保留多平台通用配置入口。
2. 一个 channel 的最小字段集合是：
   - enabled
   - name
   - webhook_url
   - secret
   - threshold
3. `secret` 是可选字段：
   - 飞书机器人未开启签名校验时，可以为空
   - 飞书机器人开启签名校验时，必须填写
4. `threshold` 的语义是“从哪个风险等级开始发消息”，而不是展示标签。
5. 测试发送功能应该保留，但它发送的是“测试消息”或“测试告警”，不是伪造整条真实 trace 告警链路。
6. 本轮不开放 payload template JSON 给用户自定义，消息模板由系统内置维护。

### 6) 工作线程数 `kernel_worker_threads`
- 价值：中高
- 成本：低
- 契合度：高
- 语义稳定性：高
- 本轮策略：做，但明确为“重启后生效”

原因：
这是明确的冷启动配置，没有必要因为不能热更新就砍掉。

当前先钉住的最小语义：

1. `kernel_worker_threads` 控制的是后端工作线程池大小。
2. 它属于冷启动配置，不做运行时动态扩缩容。
3. 用户保存后，不立即修改当前正在运行的线程池实例。
4. 这个配置只在下次启动或未来“保存并重启”流程里生效。
5. 它影响的是并发处理能力，不等于任务队列容量本身；
   不能把“worker 数量”和“队列能积压多少任务”混成同一个概念。

## 3.2 建议“本轮做简化版”的字段

### 7) 日志保留天数 `log_retention_days`
- 价值：高
- 成本：中
- 契合度：中
- 语义稳定性：中
- 本轮策略：优先考虑简化版

为什么有价值：
“只进不出”是很容易被问到的问题，尤其系统用 SQLite，本地磁盘膨胀是很现实的。

为什么不适合一口气做太满：
因为它真正涉及的是清理策略，不是一个整数输入框。

建议的简化版语义：
- 只做“按时间删”
- 每次启动时清理一次，或者定时器低频清理一次
- 统一删除超过保留期的 trace 主记录及其关联数据
- 不在第一版同时引入“按磁盘占用删”

这样做的好处是：
语义简单，容易讲清楚，而且确实能解决“数据一直长”的问题。

当前已经先钉住的最小语义：

1. 这不是“精确到点立即删除”，而是“数据一旦过期，就在下一次清理扫描时删除”。
2. 清理触发时机分两类：
   - 服务启动时先补扫一遍
   - 服务运行中由低频定时任务继续扫描
3. 如果服务中途停机，停机期间不会删除；但只要下次启动时发现数据已经超过保留期，就会在启动补扫阶段删掉。
4. 第一版删除单位按“整条 trace”处理：
   - trace 主记录
   - spans
   - analysis
   - 其他挂在这条 trace 上的关联数据
   一起删，避免留下孤儿数据。

这里先只钉“按时间删 + 启动补扫 + 运行期低频扫 + 整条 trace 级联删除”这四件事。
至于保留期最终是按 `processed_at`、落库时间，还是按 `trace_end_time` 判断，后面在真正实现前再定，不提前把细节写死。

### 8) 最大磁盘占用 `max_disk_usage_gb`
- 价值：中高
- 成本：中高
- 契合度：中
- 语义稳定性：中低
- 本轮策略：下线，不单独暴露

原因：
它当然有价值，但它必须和“删哪些数据、删到什么程度、什么时候触发”绑在一起。
如果只上这个字段，不给完整策略，反而会比没有更乱。

更合理的顺序是：
先做 `log_retention_days` 的最小闭环，再决定要不要补空间上限清理。

当前先钉住的最小语义：

1. `max_disk_usage_gb` 不适合作为一个独立字段单独留在本轮 Settings 页面里。
2. 它只有在“空间清理策略”整体成立时才有意义，
   也就是要同时回答：
   - 超限后删什么
   - 按什么顺序删
   - 与 `log_retention_days` 冲突时谁优先
   - 多久检查一次
3. 所以本轮先下线这个字段，
   不继续让它以“单独可配置”的姿态出现在 UI 里。
4. 如果以后要恢复，它应该作为“清理策略 v2 / 空间清理策略”的一部分一起回归，
   而不是自己单独回归。

### 9) AI 自动降级 `ai_auto_degrade`
- 价值：中高
- 成本：中
- 契合度：中
- 语义稳定性：中
- 本轮策略：如果做，只做极简版

建议的极简语义：
- 先走有限次数的重试，不直接一失败就降级
- 重试仍失败后，再切 fallback model / fallback provider
- fallback 目标由用户在 Settings 中显式配置，而不是代码写死
- 只处理真正的调用失败
- 不引入复杂权重、路由策略、分级切换

原因：
这个能力是有用的，但如果设计过大，马上就会把 provider 管理、错误分类和状态恢复一并卷进来。

当前先钉住的最小语义：

1. 单次失败先按“重试策略”处理，不立刻降级。
2. 重试仍失败，才进入降级。
3. 降级不是永久切换系统配置，而是“当前这次 AI 分析任务改走备用方案”。
4. 备用方案必须是用户可配的，例如：
   - 禁用降级
   - 降级到 mock
   - 降级到另一个 provider/model
5. 第一版不承诺单独做一条复杂的“重试队列”。
   语义上只要求“支持有限次数的自动重试”，实现上优先考虑在 Python AI 侧先做简单重试逻辑。

### 10) AI 熔断 `ai_circuit_breaker`
- 价值：中高
- 成本：中
- 契合度：中
- 语义稳定性：中
- 本轮策略：如果做，只做极简版

建议的极简语义：
- 连续失败超过阈值
- 暂时关闭 AI 分析
- Trace 接收/聚合/存储仍继续
- 冷却期结束后允许恢复尝试

原因：
这个能力非常适合答辩解释“系统在依赖外部 AI 服务失效时怎么降级”，但必须把语义收窄，不然很容易越做越散。

当前先钉住的最小语义：

1. 熔断的对象只是“AI 分析能力”，不是整个系统。
2. 熔断打开后：
   - 日志接收继续
   - Trace 聚合继续
   - 数据落库继续
   - Trace 查询继续
   - 只是暂停新的 AI 调用
3. 熔断和降级的顺序是：
   - 先重试
   - 再降级
   - 连续失败持续累积后才进入熔断
4. 冷却期结束后允许恢复尝试，但具体失败计数规则、恢复细节和是否半开，后面实现前再定，不在这一步提前写死。

当前先钉住的一组最小字段形态：

为了避免后面实现时再次把“重试 / 降级 / 熔断”揉成一坨，
第一版建议至少拆成下面这组字段：

1. `ai_retry_enabled`
   是否开启自动重试。

2. `ai_retry_max_attempts`
   最大重试次数。

3. `ai_auto_degrade`
   是否开启自动降级。

4. `ai_fallback_provider`
   降级目标 provider。

5. `ai_fallback_model`
   降级目标 model。

6. `ai_circuit_breaker`
   是否开启熔断。

7. `ai_failure_threshold`
   连续失败多少次后进入熔断。

8. `ai_cooldown_seconds`
   熔断后冷却多久再恢复尝试。

这一组字段已经足够形成最小闭环，
不需要在第一版继续扩展更复杂的策略参数。

## 3.3 建议“保留为规划项，暂不直接接”的字段

### 11) 国内 Provider / 新 SDK
- 价值：中高
- 成本：中高
- 契合度：中
- 语义稳定性：高
- 本轮策略：保留规划，暂不并到 Settings 收口主线

原因：
这块不是没价值，而是它本质是新 provider 接入任务，不应该和当前 Settings 真生效收口绑在一起推进。

当前先钉住的最小方向：

1. 后续优先接入的国内 provider 先定为“智谱 GLM”。
2. 接入路线优先保持和现有 `gemini` provider 一致，先走官方 Python SDK，
   不额外为了这个 provider 单独切成纯 HTTP 实现风格。
3. 这项能力仍然属于后置独立任务，
   不和当前 MVP5 的 Settings 收口主线绑在同一刀里推进。

### 12) Trace 打包/聚合控制项
包括：
- 是否通过 `trace_end` 确定打包
- token 打包限额
- span 容量打包限额
- flush / dispatch 时间阈值
- 水位阈值设置

- 价值：高
- 成本：中
- 契合度：中高
- 语义稳定性：中
- 本轮策略：值得保留，但要逐个字段重新定义语义后再接

原因：
这批字段其实很有价值，而且比“自适应模式”更贴近当前真实主链。
但问题在于它们现在还没有在 Settings 里形成一套统一、稳定、能对外讲清楚的命名和口径。

也就是说：
不是不值得做，而是应该先补“语义定义文档”，再决定先接哪几个。

当前先钉住的一个子语义：结束标记字段配置

1. 用户可以配置“哪个顶层字段表示 trace 结束”，不强制所有接入方都只能叫 `trace_end`。
2. 第一版不做 JSONPath，不支持深层嵌套路径，只处理顶层字段名或顶层别名列表。
3. 这项配置应该挂在入站协议解析层，也就是 `LogHandler` 一侧，而不是挂在 `TraceSessionManager` 状态机内部。
4. 数据流语义是：
   - Settings / ConfigRepo 提供结束标记字段配置
   - `LogHandler` 解析 JSON 时按该配置查找顶层字段
   - 命中后统一映射到标准字段 `span.trace_end`
   - `TraceSessionManager` 后续仍然只认标准化后的 `span.trace_end`
5. 这样做的目的，是把“入站协议兼容性”和“聚合状态机语义”解耦。
6. 第一版推荐形态：
   - 默认字段仍是 `trace_end`
   - 允许额外配置若干个顶层别名
   - 按顺序匹配，命中第一个合法布尔值即停止

当前先钉住的一个子语义：token 打包限额

1. `token_limit` 控制的是“单条 trace 在聚合阶段累计到多少 token 后，触发保护性封口”。
2. 它不是模型真实 usage 上限，也不是最终计费口径。
3. 它属于聚合阶段的近似预算阈值，目的是限制单条 trace 体积继续膨胀，
   同时控制后续 AI 分析负担。
4. 命中该阈值后，trace 进入 sealed / ready 流程，不再无限吸收新的 span。
5. 第一版语义里，这个 token 口径主要看 trace 自身聚合内容，
   还没有把系统固定 prompt、业务 prompt 等额外提示词开销一起算进去。
6. 所以后续真正实现或调参时，必须明确提示：
   这个阈值当前是“trace 内容侧预算”，不是“完整模型请求总 token 预算”。

当前先钉住的一个子语义：span 容量限额

1. `span_capacity` 控制的是“单条 trace 最多允许聚合多少个 span”。
2. 它属于数量维度的保护阈值，和 `token_limit` 是互补关系。
3. 命中该阈值后，也会触发保护性封口，避免单条超长 trace 持续膨胀。
4. 这项限制的主要目标，是保护聚合态内存、索引结构和后续分析成本，
   不是表示这条 trace 在业务语义上已经真正结束。
5. 所以命中 `span_capacity` 的语义是“系统为了保护自己先截断继续吸收”，
   不是“上游已经明确告诉我们 trace 完成了”。

当前先钉住的一个子语义：时间阈值组必须拆成四类

当前不再继续使用笼统的 `refresh_interval / flush_interval` 统称，
因为它会把不同生命周期阶段的时间语义混在一起。

后续真正暴露给 Settings 的时间类字段，至少要拆成下面四种：

1. 收集阶段空闲超时
   含义：
   trace 处于 collecting 状态时，多久没有收到新 span，就认为可以结束收集。

2. 封口后的乱序缓冲窗
   含义：
   命中 `trace_end / token_limit / span_capacity` 等封口条件后，
   不是立刻 finalize，而是再等一个短窗口，吸收晚到 span。

3. 分发失败后的重试等待时间
   含义：
   trace 已经 ready，但投递下游失败时，要等多久再重试。

4. 定时 sweep / tick 扫描频率
   含义：
   系统多频繁去推进超时、sealed、retry、tombstone 等状态桶。

这四类时间字段分别属于不同阶段：
- 1、2 属于收集/封口阶段
- 3 属于 ready 后的重试节奏
- 4 属于整个状态机的驱动时钟

所以后续即使最终不是四个字段全部开放给用户，也必须先按这四类语义去定义，
不能继续让一个旧名字同时承担多种时间含义。

当前先钉住的一个子语义：收集阶段空闲超时

1. 这项时间阈值只作用于 `collecting` 状态的 trace。
2. 它表达的是：
   “一条 trace 在收集阶段多久没有收到新 span，就可以按收集结束处理”。
3. 它不是 `trace_end` 的替代，而是在没有显式结束标记时的兜底结束条件。
4. 命中后，trace 进入 sealed / ready 的后续流程。
5. 它本质上是“无新数据等待多久算结束”，
   不是“整条 trace 的业务耗时上限”。

当前先钉住的一个子语义：封口后的乱序缓冲窗

1. 这项时间阈值只作用于已经命中封口条件的 trace。
2. 这里的封口条件包括：
   - `trace_end`
   - `token_limit`
   - `span_capacity`
   - 以及后续可能扩展的其他保护性封口原因
3. 命中封口后，不是立刻 finalize，
   而是再等一个短窗口，吸收少量晚到 span。
4. 这个窗口的主要目标是处理乱序，不是继续长期等待更多数据。
5. 它应该明显短于“收集阶段空闲超时”，
   因为 sealed 状态的语义已经不是“继续正常收集”，而是“准备收尾，只做短暂兜底”。

当前先钉住的一个子语义：分发失败后的重试等待时间

1. 这项时间阈值只作用于已经 ready、准备往下游 dispatch 的 trace。
2. 它的触发前提是“下游投递失败”，例如线程池 `submit` 失败。
3. 它表达的是：这次投递失败后，下一次重试前至少等待多久。
4. 它不属于 collecting 阶段超时，也不属于 sealed grace window，
   而是 ready 之后的重试节奏控制。
5. 第一版先把它定义为“指数退避的起始等待时间”，
   也就是第一次失败后的起始退避基线。
6. 后续如果要做完整指数退避，这是在这个字段之上的扩展；
   当前先不把最大退避上限、倍数策略等细节一并写死。

当前先钉住的一个子语义：sweep / tick 扫描频率

1. 这项时间阈值控制的是状态机推进的基础时钟。
2. 系统通过这个频率去扫描和推进：
   - collecting 超时
   - sealed grace window
   - ready retry
   - completed tombstone
   - 以及其他依赖时间轮 / tick 的状态
3. 它不是业务语义字段，而是更偏底层运行参数。
4. 所以即使未来开放给用户，也应该作为“高级参数”看待，
   不适合放在普通用户最先接触的一层设置里。

当前先钉住的一个子语义：水位阈值

1. 第一版仍然保留三档状态：
   - normal
   - overload
   - critical
2. 每个核心积压指标先各自只配“一组百分比阈值”，也就是对同一个指标分别给出：
   - 进入 overload 的百分比
   - 进入 critical 的百分比
3. 第一版先按“百分比”表达，而不是直接暴露原始绝对值，
   这样配置更直观，也更容易和各自容量基数绑定。
4. 这些百分比不是一个全局总水位，而是针对各自指标分别生效，
   例如：
   - active sessions 使用自己的容量基数
   - total buffered spans 使用自己的容量基数
   - pending tasks 使用线程池队列容量作为基数
5. 后端最终的背压状态，仍然由这些指标综合判定，
   不是只看某一个单独百分比。
6. 第一版先保持简单：
   - 每个指标各有一组阈值
   - 每组阈值先只包含 overload / critical 两个百分比
   - normal 由“低于 overload 阈值”自然得到，不额外单独配置

## 4. `kernel_io_buffer` 的单独评估

这项必须单独拿出来，因为它很典型：
不是“没实现”，而是“语义没钉死”。

### 4.1 价值评估
- 价值：中到中高

为什么不是低：
因为“资源预算”和“背压联动”本身是有价值的，尤其在日志/trace 聚合系统里，内存保护是迟早要回答的问题。

### 4.2 成本评估
- 成本：中到高

因为它一旦进入真实链路，就必须解决：
- 统计的到底是哪类内存
- 指标从哪里采
- 和现有三档水位如何联动
- 是用于准入门禁，还是只做告警
- 如何避免状态抖动

### 4.3 契合度评估
- 契合度：当前偏低

原因：
现有后端主链的背压，核心看的是：
- active sessions
- total buffered spans
- thread pool pending tasks

这些都是“业务积压对象”。

但 `kernel_io_buffer` 当前既没有真实采样对象，也没有真实控制对象，所以它还没法自然地挂进现有链路。

### 4.4 语义稳定性评估
- 语义稳定性：低

当前至少存在四种可能理解：

1. socket / 网络接收缓冲区预算
2. HTTP body 暂存预算
3. Trace 聚合态对象的内存预算
4. 整个进程的某种保护性内存水位

这四种语义完全不同。
如果不先钉死，后面无论做还是留，都会让 Settings 页面变脏。

### 4.5 当前结论
- 本轮策略：本轮下线，等以后以资源预算字段重设计后再回归

也就是说：
它不是低价值项，但当前名字和语义都不适合继续留在 Settings 页面里。
所以本轮先从 UI 下线，不继续以 `kernel_io_buffer` 这个名字暴露给用户。

### 4.6 如果以后要做，推荐的方向

更合理的方向不是“IO 缓冲区”，而是“聚合态/接入态资源预算”。

比如改成：
- `ingest_memory_budget_mb`
- `trace_runtime_memory_budget_mb`

然后它的使用方式应该是：
- 作为额外保护信号
- 参与抬高 overload / critical 状态
- 而不是动态放宽现有水位阈值

也就是说：
当预算占用越来越高时，系统应该更保守，而不是更激进。

## 6. 下一步建议的讨论顺序

为了避免一次聊太散，建议后面按这个顺序逐个把语义钉死：

1. 日志清理：先定“按时间删”还是“按时间+空间双策略”
2. AI 熔断/降级：先定最小状态机语义
3. 主链阈值组：先定哪些字段真的暴露给用户
4. `kernel_io_buffer`：最后再决定它是删除、改名，还是转成资源预算字段

原因：
`kernel_io_buffer` 现在不是最该先做实现的，而是最该最后做语义定稿的。
因为只要前面几组主链策略还没稳定，它就没法准确挂位。
